# 小アプリ制作テスト方針

小アプリ制作では、機能を作ったあとに「動きました」で終わらせず、最低限の自動テストを追加します。

目的はテストコードを大量に書くことではなく、要件を満たしていること、不正な入力で壊れないこと、あとから修正しても主要機能が壊れていないことを確認できる状態にすることです。

## 基本方針

- まずは正常系を 1 つ書く
- 次に失敗しやすい異常系を 1 つ書く
- DB を使うアプリでは Repository テストを 1 つ入れる
- API を作るアプリでは Controller テストを 1 つ入れる
- README に「何をテストしたか」を書く

## Java コンソールアプリ

### 入れるテスト

- 入力値を処理するロジックの単体テスト
- 計算、検索、並び替え、絞り込みのテスト
- 不正な入力を受け取った場合のテスト

### 例

- Todo を追加すると一覧件数が増える
- 存在しない ID を削除しようとすると失敗扱いになる
- 空文字のタイトルは登録できない

## Spring Boot API

### Service テスト

業務ロジックを確認します。

- Todo を作成できる
- 存在しない ID の取得で例外を投げる
- 更新時に title や description が反映される

### Controller テスト

HTTP API として正しく振る舞うかを確認します。

- `GET /tasks` が 200 を返す
- `POST /tasks` が 201 を返す
- title が空なら 400 を返す
- 存在しない ID なら 404 を返す

### Repository テスト

DB アクセスが想定通りかを確認します。

- 保存したデータを取得できる
- status で絞り込める
- title の部分一致検索ができる

## Flutter アプリ

### 入れるテスト

- widget test
- Riverpod の Provider テスト
- API 通信部分を mock したテスト

### 例

- Todo 一覧が表示される
- 入力フォームに文字を入れて追加できる
- API エラー時にエラーメッセージを表示する

## README に書くこと

小アプリごとの README には、以下を必ず書きます。

```text
## テスト

### 実行方法
npm test / mvn test / ./gradlew test など

### テストしたこと
- 正常に登録できること
- 不正な入力でエラーになること
- 存在しないデータを指定したときに404相当になること

### まだ足りないテスト
- 認証まわり
- 境界値
- 複数ユーザーのデータ分離
```

## 最初の目標

最初の Spring Boot 小アプリでは、以下の 3 種類を 1 つずつ入れることを目標にします。

- Service テスト
- MockMvc の Controller テスト
- Repository テスト

最初から全機能を網羅しようとせず、主要な CRUD のうち 1 機能を選んでテストを書きます。
